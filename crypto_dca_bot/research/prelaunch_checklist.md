# 上線前自審 Checklist — 十題(V2 backlog #2)

> **這份是什麼**：每個策略要從「模擬」走到「真錢實盤」（V2-D tiny live）之前，**最後一道人工關卡**。
> **鐵律**：下面 10 題，**任何一題答不出來 = 不准上實盤**。沒有例外、沒有「先上再補」。
> **跟主線架構（V2-A round 討論）無關**，這是一份獨立的上線檢查單。多數題目是現有 **M1–M7 驗證規格**（見 `v2_roadmap.md`）+ **V1 既有模組**的彙整濃縮 —— 真要按開關的那一刻，你看這一頁，不用回去翻整本 roadmap。

---

## 怎麼用這份清單

1. **每個策略各自過一次**，不是整個平台過一次就算數。
2. **逐題回答，答案要「具體」** —— 指得出證據（log、數字、後台截圖、測試紀錄）。不准用「應該沒問題」「大概還好」這種話帶過。
3. **任一題卡住 → 停**。回去把那題補完再來，不准跳過、不准「上了再說」。
4. 策略**邏輯改版**（比照 M3：改邏輯 = 換新編號）= 整份清單**重新過一次**。
5. 建議把每次自審的答案連同**日期 + commit hash** 存檔（比照 M3 lock 精神）。哪天出事，可以回查「當初我是憑什麼判斷它可以上的」。

**這段用人話講就是**：這是一張「上飛機前的起飛檢查表」。少檢查一項就起飛，摔下來不能怪天氣。

## 1. 這策略的市場邏輯是什麼？

**白話**：用一句人話講清楚「這策略**為什麼**會賺錢」—— 它在賺市場上**誰的、什麼錢**，而且那個錢**為什麼還沒被別人套利掉**。如果你只能說「回測數字很漂亮」但講不出「為什麼漂亮」，那大概率是運氣或過度配適（後面第 5、6 題會抓），不是真的 edge（優勢）。

**為什麼這題排第一**：沒有邏輯的策略 = 賭博。而且市場邏輯是你日後判斷「它哪天失效」的**唯一依據** —— 你要先知道它靠什麼吃飯，才知道它哪天會沒飯吃（這直接餵給 M7 退役機制）。

**怎樣算答得出來**：
- 能用**一句話**講完賺錢邏輯。範例：trend-following 賺「動能會持續一陣子」的錢；funding rate skew 賺「永續合約多頭太擁擠、資金費率過高時會回歸」的錢。
- 能說出三件事：(a) edge 來源、(b) 為什麼它還沒消失、(c) **什麼情況下它會失效**。
- 邏輯複雜到一句話講不清 → 不及格（對應簡單派定調：邏輯一句話、參數 < 5）。

**紅旗**：只會講「回測 Sharpe 很高」/ 講不出失效情境 / 邏輯要繞三層才講得完。

**這題人話 summary**：講不出「它為什麼會賺、什麼時候會不賺」，就是還沒搞懂自己在玩什麼，不准上。

## 2. 是否避開 look-ahead bias（前視偏誤）？

**先解釋名詞**：look-ahead bias = 回測時不小心**偷看了未來**。例：用「今天收盤後才知道的數字」去決定「今天開盤該不該買」—— 但實盤當下你**根本拿不到那個數字**。結果回測賺翻、實盤撲街。

**常見的偷看來源**（逐一檢查）：
- 用未來的 K 線 / 指標算當下的訊號。
- 用「整段歷史」算出的均值 / 標準差，套回過去某一天（那一天的你不可能知道未來的均值）。
- 訊號用收盤價判斷，卻假設「能用當天最低價成交」。
- 資料對齊錯位：某個指標其實是**隔天才公布**，卻拿來當「當天」的輸入。

**怎樣算答得出來**：
- 每個輸入都能回答：「這個數字在『做決定的那一刻』真的拿得到嗎？」
- 回測有 **M3 lock**：結果自動帶 timestamp + commit hash、寫進固定 log 不能事後改；策略邏輯一改就換新編號（避免 retrofit —— 事後偷調邏輯把績效弄好看）。
- M2 walk-forward 把「訓練段 / 驗證段」切開，本身就擋掉一部分前視。

**紅旗**：成交假設用了當下拿不到的價 / 指標時間軸沒對齊 / 結果可以被人事後改數。

**這題人話 summary**：回測賺錢不算數，要確定那些錢是「當下真的拿得到的資訊」賺到的，不是偷看答案。

## 3. 是否避開 survivorship bias（倖存者偏誤）？

**先解釋名詞**：survivorship bias = **只研究活下來的**。經典例：只拿「今天還在的幣」回測，但當年一大票幣早就歸零下架了 —— 你的回測自動跳過所有死掉的，於是看起來「買幣總會漲回來」，實盤買到下一個歸零的就完蛋。

**對應 V2 現況**：
- 第一階段**嚴格鎖 BTC/ETH**（roadmap 範圍鎖）。這兩個是「活下來的大幣」，本身就帶 survivorship 味道；但因為**只玩這兩個、不做選幣**，風險可控 —— 這種情況直接通過，但要在答案裡**寫明「只玩 BTC/ETH、不選幣，故不適用」**。
- **真正要警覺的是**：(a) 之後擴 Gold / Oil / NDX 或任何 altcoin 時，資料源**有沒有含已下市 / 已下架的標的**；(b) 任何「掃一籃子幣、挑強的進場」邏輯，都要問：這籃子是用「**當時實際存在的成分**」還是「**今天的成分回頭套**」。

**怎樣算答得出來**：能說清楚回測資料的「標的清單」**怎麼來的** —— 是「當時真實存在的」還是「拿今天的清單回頭套過去」。擴標的或做選幣策略前，這題**必須重答**。

**對應**：roadmap 範圍鎖（BTC/ETH）+ `data_sources.md`（資料源評估維度：歷史深度、缺漏率）。

**這題人話 summary**：別只拿「贏家名單」回測。只玩 BTC/ETH 不選幣可放行，但一碰「挑幣」或擴新市場就要重新問。

## 4. 是否含真實交易成本？

**白話**：回測有沒有把「每筆交易要付的錢」算進去 ——
- **手續費（fee）**：交易所每筆抽的。
- **滑點（slippage）**：你想用 100 買，但實際成交在 100.3，那 0.3 的價差。
- **點差 / 深度不足**：單子太大、市場吃不下，價格被你自己推走。

很多策略「帳面賺錢」純粹是因為回測**假設零成本**，把成本加回去就變賠錢 —— 尤其**頻繁調倉**的策略，最容易被手續費一口一口吃光。

**怎樣算答得出來**：
- 回測成本模型有**明確的 fee + slippage 假設**（對應 Round 1 Gap 4 成本模型題、R3-①-bc 拍板的「算量站」會做 fee / slippage 預估）。
- **M5**：paper vs backtest 的 **fill rate（成交率）差 ≤ 10%** —— 成交率差太多，代表回測對「能不能成交」的假設不貼實盤。
- 理想：已過 **tiny live（V2-D step 4，50–100 USDT）**，用真錢量過真實 slippage + fee + 心理。

**紅旗**：回測零成本 / 成本只是「猜一個固定 bps」沒驗過 / 高頻調倉卻沒壓測成本。

**這題人話 summary**：把手續費跟滑點加回去後還賺，才算真的賺。零成本回測的漂亮數字不算。

## 5. 參數是否穩定？

**先解釋名詞**：**參數** = 策略裡可以調的旋鈕（例：均線取 20 天還是 50 天、進場門檻設多少）。**參數穩定** = 你把旋鈕稍微轉一點（20 天改 22 天），績效不會從賺錢瞬間變賠錢。

如果一定要剛好 20 天才賺、19 跟 21 都賠 —— 那 20 是**湊出來的**（overfit / 過度配適：把策略硬調到剛好符合過去那段歷史，但**過去不會原封不動重演**）。

**怎樣算答得出來**：
- 做過**參數敏感度測試**：目標參數**附近一圈**的績效都還行（一片平原），不是孤零零一個尖峰。
- **參數數量少**（簡單派：理想 < 5 個。旋鈕越多，越容易把歷史背起來）。
- M2 walk-forward 每次重訓選出的參數**不會劇烈跳動**（這次選 20、下次跳到 80，代表不穩）。

**紅旗**：參數一動就崩 / 參數 > 5 個還在加 / 「我們已經調到最佳參數了」（**最佳 = 最可疑**，通常就是 overfit 的鐵證）。

**這題人話 summary**：好策略是「一片平原」，差一點也還行；overfit 是「一根針尖」，差一點就掉下去。站在針尖上不准上。

## 6. Out-of-sample 是否還活著？

**先解釋名詞**：**out-of-sample（樣本外，簡稱 OOS）** = 把歷史切兩段：前段拿來「訓練 / 調參數」（叫 **in-sample 樣本內**），後段**藏起來完全不給策略看**；等策略調好了，才拿後段來「考試」。如果策略只在它見過的前段賺、一到沒見過的後段就死 —— 它只是**背答案**，不是真懂。

**怎樣算答得出來（直接抄 M2 寫死規格）**：
- **IS（訓練）30 個月 / OOS（驗證）3 個月**。
- **每 3 個月重訓**一次（walk-forward 滾動）。
- **WFE > 50%**。WFE（walk-forward efficiency，樣本外效率）= 樣本外績效 ÷ 樣本內績效。低於 50% = 一到沒見過的資料就掉一半以上，等於背答案。
- **OOS 每個視窗成交 ≥ 30 筆** —— 太少筆數，統計上不算數，賺賠都可能只是運氣。

**紅旗**：只有 in-sample 數字漂亮 / WFE < 50% / OOS trade 數 < 30。

**這題人話 summary**：把後段歷史藏起來考它，考過（WFE > 50%、樣本夠多）才算真懂，不是背答案。

## 7. Paper trading 是否正常？

**先解釋名詞**：**paper trading（紙上交易 / 模擬下單）** = 用**真實即時行情**但**假錢**跑策略。回測是「歷史靜態資料」，paper 是「活的、會跳動的市場」—— 看它在真實節奏下會不會做蠢事。這是上真錢前最後一道**免費**防線。

**怎樣算答得出來（抄 M4 + M5）**：
- **至少跑滿 60 個交易日**（trading day，週末 / 假日不算）。為什麼是交易日不是自然日？要讓它至少遇到 1–2 次**月度市況轉變**（regime shift）。
- **paper vs backtest 並排比較**：
  - **Sharpe 差 ≤ 30%**。Sharpe（夏普比率）= 衡量「每承擔一單位波動，換到多少報酬」的 CP 值指標。差太多 = 模型有結構性誤差。
  - **fill rate（成交率）差 ≤ 10%**。
  - 任何一項超標 = reject，**不上真錢**。

**紅旗**：paper 沒跑滿 60 交易日就想上 / paper 跟 backtest 差很多卻想「應該還好」硬上。

**這題人話 summary**：用假錢在真市場跑滿 60 個交易日，而且表現要對得上回測；對不上就是模型有鬼，不准上。

## 8. 最大虧損是否可接受？

**先解釋名詞**：**最大回撤（max drawdown）** = 帳戶從最高點跌到最低點，中間**最深跌了幾 %**。這是「最痛的那一刻」。你要先誠實問自己：「真的跌這麼多，我半夜扛得住、不會自己手動關掉嗎？」—— **扛不住的策略，再賺也會被你自己在最低點砍掉**。

**怎樣算答得出來**：
- (a) **過 M1 五段歷史崩盤壓力測試**：COVID 2020-03 / China 2021-05 / LUNA 2022-05 / FTX 2022-11 / 日圓 carry unwind 2024-08。**任何一段過不了 = reject**。
- (b) **M6 部位大小是 risk-based**（風險基礎，不是固定比例 / 永遠滿倉）：預設 volatility targeting（波動越大、部位自動越小）。而且 risk-based 版的 max drawdown 要比 naive 固定版**低 ≥ 20%**，否則 sizing 設計無效，回去改。
- (c) 上線後有 **M7 回撤熔斷**：live 最大回撤超過 backtest 的 **1.5 倍** → 立即縮到 0 部位 + 告警。
- (d) 還有 **Risk Engine**（V2 第三個一級元件，framework 級護欄、always-on 不可關）的 **portfolio-gross 總曝險上限**兜底。
- (e) 你心裡有一個**明確數字**：「跌到 X% 我接受，超過就停。」（V1 `high_water_mark.py` 已有追蹤帳戶歷史新高的基礎件，回撤就是從這個高點往下量。）

**紅旗**：沒跑過 M1 崩盤段 / 固定滿倉 / **講不出自己能接受的最大跌幅數字**。

**這題人話 summary**：先想清楚「最慘會跌多少、我扛不扛得住」，過了五段歷史崩盤、部位會隨波動自動縮、跌太兇會自動熔斷 —— 三道都備齊才放行。

## 9. API key 是否無提款權限且有 IP whitelist？

**白話**：這題跟賺不賺錢無關，是「**別人偷不偷得走你的錢**」。交易所的 API key 就像一把帳戶鑰匙。萬一被偷（電腦中毒、key 外洩、不小心 git commit 上去），後果分兩個等級：
- 只開「**交易 + 讀取**」：頂多被亂買亂賣，**錢還在交易所**，還救得回來。
- 開了「**提款 / withdraw**」：直接把幣**轉走**，錢沒了，救不回來。

所以鐵律兩條：
1. 這把 key **只開「交易 + 讀取」，絕對不開「提款 / withdraw」**。
2. 綁 **IP whitelist（白名單）** —— 只准你那台主機的 IP 用這把 key，別人就算偷到也用不了。

**怎樣算答得出來**：
- 去交易所後台**看 key 的權限頁親眼確認**：withdraw 是**關**的、IP 白名單有填且**只有你的主機 IP**。
- V1 `exchange_api.py` 本來就是這個設計（docstring 寫明 *"Reading permission only, IP-whitelisted"*，且 code 對 `PermissionDenied` 會提示「檢查 IP 白名單或 API key 權限設定」）。**V2 沿用 `exchange_api.py` 必須維持這個前提。**
- key **不進 git**（`.gitignore` 已擋 `.env`）、**不寫死在 code**、**不出現在 log**。

**紅旗**：key 開了提款權限 / 沒設 IP 白名單 / key 出現在 git 歷史或 log 裡。

**這題人話 summary**：鑰匙只給「能買賣、不能搬錢」的權限，而且只認你家門牌（IP）。這題是binary —— 沒做到就是不准上，不囉嗦。

## 10. 系統失控時是否能自動停機？

**白話**：程式**一定**會出包（網路斷、交易所改 API、策略踩到沒想到的狀況）。問題不是「會不會出包」，而是「出包時它會**自己停下來**，還是會在你睡覺時**瘋狂亂下單把錢燒光**」。24h 全自動跑的系統，**沒有自動停機 = 一顆定時炸彈**。

**怎樣算答得出來（V2 要有的幾層，多數 V1 已有、V2 沿用 + 強化）**：
- **連敗熔斷**：連續 N 次失敗 → 推播 + **直接關掉程式**（V1 `circuit_breaker.py`：`sys.exit(1)`，exit code 讓排程器看到「失敗」而不會盲目重啟）。
- **心跳（heartbeat）**：每 6h 發一次存活訊號（V1 `heartbeat.py`）。太久沒訊號 = 出事了你會知道，防「**靜默死亡**」（程式 crash 但沒人發現，還以為在跑）。
- **回撤熔斷**：M7，live 回撤超 backtest 1.5 倍 → 縮 0 倉 + 告警。
- **策略缺席 / crash 湧現停機**：Round 2 #2D + #3A —— 策略連續 crash N 次永久停用；守門員（PortfolioStrategy）全倒光 → 撞 always-on 鎖 → framework **自動整台停機**。
- **Risk Engine always-on 總曝險上限**：framework 級護欄，**不可關**，永遠在。
- **而且這些自動停機必須被驗證過真的會動** —— V1 `chaos_test.py`（失敗注入測試）就是專門證明「踩到地雷時它真的會停」。**上線前必須跑過一次 chaos test。**

**紅旗**：根本沒有 kill switch / 有但**從沒測過**會不會動 / crash 後會**無腦自動重啟繼續下單**。

**這題人話 summary**：出包時要會「自己拔插頭」，而且這個拔插頭的動作**要事先用 chaos test 證明過真的會拔**，不是寫在那裡好看。

## 通過標準（總則）

- **10 題全部能「具體」回答**（指得出證據：log / 數字 / 後台截圖 / 測試紀錄），才放行上 **V2-D tiny live**。
- 任一題答「**不知道 / 還沒做 / 應該沒問題**」= **不准上實盤**。沒有例外，沒有「先上再補」。
- **每個策略各自過一次**；策略改版（M3 新編號）= 重新過一次。
- 把每次自審的答案連同**日期 + commit hash** 存檔（比照 M3 lock）。出事可回查當初的判斷。

**整份人話 summary**：這 10 題就是「能不能讓它碰真錢」的最後守門。第 1–8 題確認「策略本身是真的、不是回測幻覺」，第 9–10 題確認「就算出事，錢搬不走、系統會自停」。**任何一題卡住，停在原地補，不准放它上線。**
